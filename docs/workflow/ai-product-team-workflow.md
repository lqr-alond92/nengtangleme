# AI 产品团队工作流

本文档定义「能躺了吗」项目的产品协作机制。后续任何功能进入实现前，都必须先完成角色拆解、方案审查、工程评估、QA 验收标准，并由 CEO 明确批准。

## 角色边界

### CEO

CEO 负责提出目标、判断方向、做最终取舍。

CEO 需要拍板的内容包括：

- 功能是否进入本轮版本
- 产品叙事和用户价值是否成立
- 关键页面结构和主文案方向
- AI 能力边界和风险表达尺度
- 是否允许进入代码实现

### 项目负责人

项目负责人负责组织角色协作、汇总冲突、守住流程。

项目负责人不直接跳过流程写代码；当 CEO 提出需求后，必须按固定顺序输出：

1. CEO Brief
2. PM 拆解
3. UX Review
4. Engineering Plan
5. QA Plan
6. CEO 待拍板事项

### PM

PM 负责把目标拆成 3-5 个可实现任务。

PM 输出必须包括：

- 功能范围
- 不做范围
- 优先级
- 里程碑
- 版本影响
- 与现有 PRD 的差异

### UX

UX 负责检查用户流程、页面状态和文案。

UX 输出必须包括：

- 用户从哪里进入
- 每一步看到什么
- 每一步能做什么
- 空状态、加载态、错误态、完成态
- 关键文案是否清楚、有 AI 感、不过度刺激
- 是否一次抛出过多信息

### 前端工程师

前端工程师只负责前端实现方案和前端代码。

前端工程师输出必须包括：

- 页面和组件拆分
- 状态流转
- 输入组件
- 动效和响应式
- 本地缓存和兼容问题
- 实现风险

### 后端工程师

后端工程师只处理数据、接口、模型调用和服务端逻辑。

后端工程师输出必须包括：

- API 边界
- AI 模型接入方案
- Prompt 输入结构
- 数据持久化
- 鉴权和成本控制
- 失败降级方案

### QA

QA 负责最后验收，列出失败点和需要修复的问题。

QA 输出必须包括：

- 测试点
- 边界情况
- 验收标准
- 失败点
- 必须修复项
- 可延期项

## 标准协作流程

每次功能开发按以下顺序执行：

1. CEO 提出目标
2. 项目负责人复述 CEO Brief
3. PM 拆解任务
4. UX 检查流程和文案
5. 前端工程师给出前端实现方案
6. 后端工程师给出数据和接口方案
7. QA 给出验收标准和测试清单
8. 项目负责人汇总冲突和待拍板问题
9. CEO 批准
10. 进入实现
11. 跑测试、构建、预览验证
12. QA 输出验收结果
13. 更新 PRD、决策记录和版本变更

## 强制门禁

以下情况禁止进入代码实现：

- 没有 CEO Brief
- PM 没有拆出任务
- UX 没有检查流程和文案
- 工程师没有说明实现方案
- QA 没有验收标准
- CEO 没有明确批准
- PRD 没有版本记录或差异说明

## PRD 管理规则

每项产品方案必须有对应 PRD 记录。

PRD 存放在：

```text
docs/prd/PRD-xxx-feature-name/
```

每次重要调整必须新增版本，不直接覆盖旧版本：

```text
v1.0.md
v1.1.md
v1.2.md
CHANGELOG.md
decisions.md
```

版本规则：

- `v0.x`：早期草案
- `v1.0`：首次确认版本
- `v1.x`：同一功能的迭代优化
- `v2.0`：核心流程、数据结构或产品定位发生重大变化

## 决策记录规则

凡是 CEO 明确拍板的关键事项，必须写入 `decisions.md`。

记录格式：

```text
日期：
决策：
背景：
放弃的方案：
影响范围：
后续观察：
```

## 实现后验收规则

实现完成后必须至少完成：

- `npm test`
- `npm run build`
- 本地预览可访问
- 关键流程人工走查
- 移动端布局检查

没有完成验证前，不能声明功能完成。

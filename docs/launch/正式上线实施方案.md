# 能躺了吗正式上线实施方案

版本：v0.1  
日期：2026-06-05  
状态：Draft  
目标：把当前 PWA 原型升级为可登录、可保存、可接入通义千问、可上线内测的正式 Web App。

## 1. 总体目标

当前项目已经完成了家庭财务规划 MVP 原型，包括问诊、资产、目标、规划结果页和 AI 补充入口。下一阶段要从“本地原型”进入“正式上线产品”。

上线目标不是一次性做成完整金融 App，而是先做一个可控的 Web App 内测版：

1. 用户可以注册 / 登录；
2. 用户的资产、负债、收入、支出、目标和 AI 对话可以保存；
3. 本地计算逻辑迁移到后端，前端只负责展示和交互；
4. 接入通义千问，负责结果页解释和继续提问；
5. AI 输出有安全模板，不提供投资建议，不承诺收益；
6. 具备隐私政策、用户协议、免责声明和模型公示；
7. 部署到中国内地生产环境，准备 ICP 备案和后续 App 备案。

## 3. 推荐技术架构

### 3.1 当前架构

```text
React 前端
  ↓
浏览器 localStorage
  ↓
本地财务计算
```

问题：

- 数据只在用户本机；
- 换设备无法继续；
- 模型 Key 不能放前端；
- AI 对话没有服务端日志；
- 无法做用户权限、限流、风控和合规留痕。

### 3.2 正式上线架构

```text
React Web App / PWA
  ↓ HTTPS
Backend API
  ↓
PostgreSQL
Redis
Tongyi Qianwen / Alibaba Cloud Model Studio
Log & Monitoring
```

职责拆分：

- 前端：
  - 登录态展示；
  - 问诊交互；
  - 资产 / 目标 / 规划页展示；
  - AI 对话 UI；
  - 隐私政策和用户协议页面。

- 后端：
  - 鉴权；
  - 数据保存；
  - 财务计算；
  - 模型调用；
  - AI 输出校验；
  - 限流；
  - 日志；
  - 数据导出 / 删除。

- 数据库：
  - 用户；
  - 家庭计划；
  - 资产快照；
  - 目标；
  - AI 对话；
  - 模型调用记录；
  - 用户反馈。

## 4. 后端 API 设计

### 4.1 鉴权 API

```text
POST /api/auth/send-code
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/me
POST /api/auth/delete-account
```

说明：

- 第一版建议用手机号验证码或邮箱验证码；
- 登录成功后使用 HttpOnly Cookie 或短期 token；
- 账号注销必须能删除或匿名化用户数据。

### 4.2 计划 API

```text
GET    /api/plans/current
POST   /api/plans
PUT    /api/plans/:planId
DELETE /api/plans/:planId
```

保存字段：

- 可立即动用资产；
- 不易变现资产；
- 负债；
- 年收入；
- 年支出；
- 可工作年限；
- 收益率；
- 通胀率；
- 折现率。

### 4.3 目标 API

```text
GET    /api/plans/:planId/goals
POST   /api/plans/:planId/goals
PUT    /api/plans/:planId/goals/:goalId
DELETE /api/plans/:planId/goals/:goalId
```

目标类型：

- 一次性目标；
- 持续性目标；
- 必须目标；
- 想要目标。

### 4.4 计算 API

```text
POST /api/plans/:planId/calculate
GET  /api/plans/:planId/report
```

说明：

- 当前 `src/finance.mjs` 的计算逻辑迁移到后端共享模块；
- 前端不再自己作为唯一计算来源；
- 后端返回结构化 `metrics`、`ratios`、`risks`、`judgement`。

### 4.5 AI API

```text
POST /api/ai/explain-report
POST /api/ai/continue-chat
POST /api/ai/supplement-plan
GET  /api/ai/conversations/:conversationId
```

用途：

- `explain-report`：生成结果页解释；
- `continue-chat`：用户继续问 AI；
- `supplement-plan`：用户补充资产或目标；
- `conversations`：读取历史对话。

### 4.6 反馈 API

```text
POST /api/feedback
```

反馈类型：

- 结果不准确；
- AI 回答看不懂；
- 数据填写困难；
- 隐私或删除诉求；
- 其他建议。

## 6. 通义千问接入方案

### 6.1 接入方式

建议使用阿里云百炼 / DashScope 的 OpenAI 兼容接口。

后端保存：

```text
QWEN_API_KEY
QWEN_BASE_URL
QWEN_MODEL
```

前端绝不保存模型 Key。

### 6.2 模型选择策略

第一轮测试：

- 质量优先：选更强的通义千问模型做结果解释；
- 成本优先：选较轻模型做继续问答；
- 后端保留模型配置开关，方便切换。

最终选择指标：

- 中文表达是否自然；
- 是否能稳定输出 JSON；
- 是否会给出投资建议；
- 响应速度；
- 单次成本；
- 失败率。

### 6.3 AI 输入模板

后端传给模型的输入必须包含：

```text
用户计划摘要
关键指标 metrics
风险列表 risks
目标列表 goals
安全规则
输出 JSON Schema
```

禁止把无关日志、手机号、邮箱等个人身份信息传给模型。

### 6.4 AI 输出 JSON Schema

```json
{
  "leadIn": "string",
  "verdict": "string",
  "reason": "string",
  "keyMetrics": [
    {
      "label": "string",
      "value": "string",
      "note": "string"
    }
  ],
  "pressure": {
    "title": "string",
    "evidence": "string",
    "impact": "string"
  },
  "longTermAdvice": {
    "target": "string",
    "recommendation": "string",
    "impact": "string"
  },
  "shortTermPlan": [
    {
      "label": "string",
      "value": "string",
      "note": "string"
    }
  ],
  "disclaimer": "string"
}
```

### 6.5 AI 安全规则

模型必须遵守：

- 不推荐具体股票、基金、保险、理财产品；
- 不承诺收益；
- 不使用“保证”“一定”“稳赚”等表达；
- 不替用户做不可逆财务决定；
- 不鼓励用户立即辞职、断供、借贷投资；
- 所有结论都基于用户输入和假设；
- 必须提示“非投资建议”；
- 数据不足时必须要求用户补充信息。

### 6.6 失败兜底

如果模型失败：

- 返回本地规则生成的 `buildAiReport`；
- 页面提示“AI 解释暂时不可用，已展示本地测算结果”；
- 记录错误日志；
- 不阻断用户查看规划页。

## 7. 合规与文案

### 7.1 必备页面

```text
/terms          用户协议
/privacy        隐私政策
/ai-disclaimer  AI 使用说明与非投资建议
/model-info     模型服务公示
/delete-account 账号注销与数据删除说明
/contact        联系与投诉
```

### 7.2 用户协议要点

- 产品定位：家庭财务规划测算工具；
- 不是金融投资顾问；
- 不提供证券、基金、保险等具体产品推荐；
- 用户需对输入数据真实性负责；
- 测算结果只供参考；
- AI 回答可能有误；
- 用户应结合自身情况和专业意见决策。

### 7.3 隐私政策要点

必须清楚说明：

- 收集什么信息；
- 为什么收集；
- 如何使用；
- 保存多久；
- 是否传给第三方模型服务；
- 用户如何删除；
- 用户如何注销；
- 如何联系产品方。

特别说明：

- 资产、负债、收入、支出、家庭目标属于敏感度很高的信息；
- 必须采用最小必要原则；
- 不需要的信息不要收集；
- 不要把手机号、邮箱等身份信息传给模型。

### 7.4 AI 使用说明

需要写明：

- 本产品使用大模型生成解释；
- AI 输出基于用户输入和系统假设；
- AI 可能出错；
- AI 不构成投资、税务、法律、保险建议；
- 用户可以要求删除 AI 对话记录；
- 产品会对模型输出做安全限制。

### 7.5 模型公示

设置页或产品底部需要展示：

```text
本产品 AI 能力由通义千问相关模型提供。
具体模型名称、服务提供方、备案信息以上线时实际配置为准。
```

上线前需要补充准确模型名称和备案信息。

## 10. 开发里程碑

### 阶段 0：产品冻结

目标：确定内测版到底做什么。

产物：

- 本文档确认；
- PRD v1.5；
- 页面范围清单；
- 不做事项清单。

完成标准：

- 不再临时加大功能；
- 明确第一版只做 Web App 内测。

### 阶段 1：后端基础

目标：把本地 App 变成有服务端的 App。

任务：

- 创建后端项目；
- 建数据库；
- 建用户表；
- 建计划表；
- 建目标表；
- 建登录接口；
- 建计划保存接口。

完成标准：

- 用户可以登录；
- 刷新页面后数据还在；
- 换浏览器登录后能看到同一份计划。

### 阶段 2：计算迁移

目标：把财务计算从纯前端迁到后端。

任务：

- 抽取 `finance.mjs` 为共享计算模块；
- 后端提供 `/calculate`；
- 前端改为调用后端；
- 保留前端兜底计算。

完成标准：

- 同一份输入，前后端结果一致；
- 测试覆盖核心计算。

### 阶段 3：通义千问接入

目标：AI 解释和继续问 AI 可用。

任务：

- 后端接入通义千问；
- 编写 Prompt；
- 定义 JSON Schema；
- 做输出校验；
- 做失败兜底；
- 保存 AI 对话记录。

完成标准：

- 结果页解释来自后端 AI；
- AI 不输出具体投资建议；
- 模型失败时页面仍可用。

### 阶段 4：合规页面

目标：让内测版具备基本上线材料。

任务：

- 用户协议；
- 隐私政策；
- AI 使用说明；
- 非投资建议免责声明；
- 模型服务公示；
- 账号注销说明；
- 联系与投诉页面。

完成标准：

- 页面底部能访问全部文档；
- 登录前能看到隐私政策；
- 用户能找到注销说明。

### 阶段 5：内测部署

目标：让 20-50 人真实使用。

任务：

- 购买域名；
- 部署 staging；
- 配置 HTTPS；
- 配置数据库备份；
- 配置日志和错误监控；
- 邀请内测用户。

完成标准：

- 内测用户可以注册、填写、保存、继续问 AI；
- 收集至少 20 条真实反馈；
- 知道每次 AI 调用成本。

### 阶段 6：备案与正式发布

目标：进入正式可访问状态。

任务：

- ICP 备案；
- 公安联网备案；
- 更新页脚备案号；
- 根据实际模型补全模型公示；
- 修复内测问题；
- 发布 production。

完成标准：

- 域名可访问；
- 页面有备案号；
- 合规页面齐全；
- 错误率和模型成本可控。

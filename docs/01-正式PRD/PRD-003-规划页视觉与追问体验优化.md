# PRD-003 规划页视觉与追问体验优化

状态：Draft / 已进入 CloudBase Demo 验证  
日期：2026-06-14  
适用版本：CloudBase Demo / Mobile Alpha  
相关原型：`docs/04-原型与工作流/result-page-ai-rework-demo.html`  
相关文档：`PRD-002-AI结果页与DeepSeek结构化输出.md`

## 1. 背景

当前版本已经完成静态网站上传、DeepSeek 云函数代理接入、AI 结果页基础结构化输出和追问能力。但实际手机浏览器体验中出现了几个明显问题：

1. 结果页视觉和原型之间存在差距，字体大小、颜色、边框、卡片层级不统一。
2. 结果页存在重复表达和“框里套框”，用户难以抓住核心结论。
3. 追问回答曾直接展示 DeepSeek 的 Markdown 长文本，导致 `**`、编号、长段落等内容在手机上显得混乱。
4. 首页和问诊页仍带有“手机样机”式外框，真实手机访问时形成“手机里再套手机”的效果，顶部和两侧浪费大量可阅读空间。

本 PRD 记录这一轮已确认的优化方向，作为后续开发和验收依据。

## 2. 产品目标

让用户在手机上进入「能躺了吗」后，获得一个更像真实 App、而不是原型展示页的体验：

1. 页面首屏直接进入产品内容，不再展示多余手机样机外框。
2. AI 分析结果按固定信息结构呈现，但具体文案和判断来自 DeepSeek。
3. 追问回答保持结构化、短句化、卡片化，避免长段 Markdown。
4. 所有核心模块统一字体、颜色、边框和信息密度。

## 3. 用户流程

### 3.1 首次进入

用户打开链接后，直接看到「能躺了吗？AI 在线问诊」的产品内容。

要求：

1. 页面内容从手机浏览器可视区顶部开始。
2. 不再使用居中的手机样机外壳。
3. 不再出现顶部大面积浅色留白。
4. 背景、圆角、阴影只服务于内容，不制造额外容器感。

### 3.2 信息输入

用户通过问诊方式输入家庭画像、资产、负债、收入、支出和未来目标。

已确认方向：

1. 资产类问题不要无限拆多轮，但也不能把所有资产、负债、收入、支出塞进同一个视野。
2. 每个问题模块控制信息密度，让用户一次回答同类问题。
3. 勾选或输入问题时，当前问题应常驻在底部，避免用户看不到正在回答什么。

### 3.3 AI 分析生成

用户完成输入后，系统调用 DeepSeek 生成结构化分析。

要求：

1. 先出现文案，再出现对应可视化卡片。
2. 不向用户暴露模型名、接口名、结构化生成过程。
3. Loading 使用更自然的分析状态，例如“正在整理你的家庭财务地图”。

### 3.4 结果阅读

结果页按照以下顺序展示：

1. 分析说明
2. 重构后的真实资产
3. 核心结论
4. 风险
5. 行动建议
6. 底部追问输入框

底部追问输入框常驻，用户随时可以继续问。

追问输入框上方保留轻量工具按钮，用 icon 承载“更新资产结构”“新增一个目标”等高频动作；不再使用大卡片入口，避免挤占阅读空间。

结果页分析生成时不要每完成一个模块就强制滚到底部。文字和卡片按顺序出现即可，用户阅读位置不应被系统频繁打断；只有进入追问聊天流后，聊天区可以自动跟随最新消息。

## 4. 模块规范

### 4.1 顶部与导航

当前 Demo 不需要复杂导航。

已确认：

1. 删除顶部的“知心姐姐 / DeepSeek / AI 已连接”等技术或人格标签。
2. 删除悬浮的“AI 监测”按钮。
3. 删除底部“资产 / 规划 / 目标 / 陪跑 / 模型”导航。
4. 只保留一个“规划”主界面。
5. 左上角保留 `...` 菜单按钮。
6. 点击 `...` 后弹出菜单，包含：
   - 资产
   - 目标
   - 计划

### 4.2 重构后的真实资产

模块目标：让用户知道后面所有判断到底用哪两个钱数。

展示两张同级大卡，不再使用外层绿色框，也不再在大框里继续套小框。

第一张卡：

```text
合起来能规划的钱
1,056 万

这笔钱表示还没扣负债前，当前和未来合起来能规划的总额，由当前可动用资产 15 万 + 未来收入今天价值 1,041 万组成。
```

第二张卡：

```text
扣掉已有负债后，真正能用于目标的钱
606 万

这笔钱表示扣掉已有负债后，真正能用于未来目标的金额，由 1,056 万 - 当前总负债 450 万得到。
```

视觉要求：

1. 只用统一字体和统一说明文字颜色。
2. 说明文字不要加粗，不要单独上色。
3. 债务信息作为第二张卡的说明，不作为中间大卡。
4. 不展示复杂公式，只在说明中自然表达计算关系。

### 4.3 核心结论

模块目标：告诉用户目标覆盖状态，而不是重复资产大数。

核心逻辑分三种状态：

#### 状态 A：全部目标都能完成

展示：

1. 主标题：全部目标可以覆盖。
2. 目标折现总额。
3. 必要目标列表，每个目标显示名称、折现值、对号。
4. 想要目标列表，每个目标显示名称、折现值、对号。
5. 覆盖后盈余。

#### 状态 B：必要目标能完成，想要目标不能完成

展示：

1. 主标题：先保必要目标，再重排想要目标。
2. 必要目标折现总额。
3. 必要目标列表，每个目标显示名称、折现值、对号。
4. 想要目标折现总额。
5. 想要目标列表，每个目标显示名称、折现值、问号。
6. 扣掉必要目标后的余量。
7. 想要目标总缺口。

原则：

1. 不替用户决定哪个想要目标先被砍掉。
2. 不给每个想要目标单独判断是否达成。
3. 只把想要目标当作一组“选择券”，告诉用户总缺口。

#### 状态 C：必要目标也不能完成

展示：

1. 主标题：必要目标也存在缺口，需要先降低风险。
2. 必要目标折现总额。
3. 必要目标列表，每个目标显示名称、折现值、问号。
4. 必要目标缺口。
5. 想要目标折现总额。
6. 想要目标列表，每个目标显示名称、折现值、问号。

标识规则：

1. 能覆盖的目标用对号。
2. 超出当前可规划能力的目标用问号。
3. 只有对号和问号两种状态，不用复杂颜色表达。

### 4.4 风险

模块目标：解释为什么会有风险。

展示方式：

1. 统一一个简洁外框。
2. 每个风险只包含：
   - 风险内容是什么
   - 原因是什么
3. 不在风险模块里写行动手段。
4. 不使用多种底色、图表或复杂拆解。

示例：

```text
灵活资产严重不足
原因：当前可动用资产只有 15 万，低于 6 个月应急储备标准。遇到失业、疾病或家庭突发支出时，容易被迫借钱或出售资产。
```

### 4.5 行动建议

模块目标：把分析变成用户能点、能设置、能修改的后续动作。

结构：

1. 横向滑动卡片。
2. 每张卡只保留三层：
   - 标题
   - 原因
   - 目标
3. 按钮直接表达动作，例如：
   - 设置计划
   - 调整目标
   - 建立账户

不要出现：

1. “行动类行动”
2. “复合输入”
3. 散落的标签信息
4. 多种颜色混在同一张卡里

计划弹窗要求：

1. 展示当前状态。
2. 展示目标状态。
3. 展示月度动作。
4. 展示周度动作。
5. 展示预计周期。
6. 生成对应计划清单。

### 4.6 追问回答

追问回答必须结构化展示，不允许直接渲染 DeepSeek 长段 Markdown。

DeepSeek 返回要求：

```json
{
  "headline": "一句话回答用户这次问题，18 个中文以内",
  "summary": "用 1-2 句话解释判断，必须引用关键数字，80 个中文以内",
  "cards": [
    {
      "title": "短标题",
      "body": "具体判断或建议，60 个中文以内"
    }
  ],
  "nextQuestions": [
    "用户可能会继续这样问，必须是第一人称用户口吻"
  ]
}
```

前端兜底要求：

1. 如果模型没有返回合法 JSON，前端要清洗 Markdown。
2. 去掉 `**`、长编号、代码块等符号。
3. 自动拆成最多 3 张卡片。
4. 下一步追问最多 3 个。
5. 下一步追问必须转成用户口吻，例如“我先保必要目标，想要目标还能剩多少？”，不能写成“你是否需要……”。
6. 更新资产、新增目标等补充动作固定在输入框上方，以 icon-only 小按钮呈现，并保留 `aria-label` / `title` 说明。

## 5. 视觉规范

整体原则：

1. 字号统一，不在同一模块里频繁跳级。
2. 颜色克制，主色只用于核心数字、状态标识和主要按钮。
3. 边框、圆角、阴影统一。
4. 不做框里套框。
5. 不用装饰性图表把简单问题复杂化。
6. 移动端优先，375px 宽度下不得出现核心内容被挤压、遮挡或横向溢出。

## 6. DeepSeek 要求

DeepSeek 负责：

1. 根据用户输入生成真实资产解释文案。
2. 判断目标覆盖状态。
3. 输出核心结论和处境洞见。
4. 输出风险内容和原因。
5. 输出行动建议的原因和目标。
6. 根据上下文生成追问回答和下一步问题。

DeepSeek 不负责：

1. 决定视觉样式。
2. 输出 Markdown 报告。
3. 输出固定规则文案。
4. 暴露模型名、接口名或技术过程。

## 7. 验收标准

### 7.1 移动端空间

1. 手机真实浏览器中，页面内容应从浏览器工具栏下方直接开始。
2. 不出现“手机壳套手机壳”的样机感。
3. 页面左右不应有明显大留白。
4. 首页、问诊页、结果页都应使用同一套移动端容器策略。

### 7.2 结果页

1. 重构后的真实资产只展示两张同级大卡。
2. 核心结论按三种目标覆盖状态正确展示。
3. 风险模块只展示风险和原因。
4. 行动建议卡片结构统一。
5. 不出现多种不必要底色、边框和字体层级。

### 7.3 追问

1. 追问回答不展示 Markdown 符号。
2. 回答有标题、摘要、卡片和下一步追问。
3. 底部输入框常驻。
4. 用户可以连续追问，追问聊天区自动跟随最新回复；结果页初次分析生成不强制滚到底部。

## 8. 已知问题与下一步

当前首页留白的主要原因是线上仍复用了原型阶段的 `onboarding-phone` 手机样机容器。下一步需要把首页、问诊页、Loading 页从“样机展示模式”切换为“真实手机全屏模式”：

1. `onboarding-shell` 取消外层居中展示和上下 padding。
2. `onboarding-phone` 改为全宽、全高、无外框、无大圆角。
3. 删除或弱化 `phone-grip`。
4. 压缩 hero 顶部间距。
5. 在真实手机和浏览器模拟器中分别验收首屏可读面积。

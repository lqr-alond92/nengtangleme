# 能躺了吗产品迭代双线方案

当前版本已经完成手机可访问 Demo、CloudBase 静态托管和 DeepSeek 云函数接入。下一阶段不急着扩功能，先围绕两条主线优化体验闭环：

1. 从首页到用户输入：让用户第一眼看懂，愿意填，并且填得顺。
2. AI 分析输出：减少固定规则感，让大模型基于用户画像生成真正有差异的判断。

## 总目标

让体验用户在手机上完成一条清晰链路：

打开链接 -> 明白产品用途 -> 输入家庭财务信息 -> 得到 AI 分析 -> 继续追问或修正计划。

本阶段仍然保持：

- 不登录、不注册。
- 用户规划数据保存在当前设备浏览器。
- DeepSeek Key 只放在 CloudBase 云函数环境变量。
- 前端不展示或要求体验用户填写模型 Key。

## 分支一：首页到输入

### 目标

让用户不用解释也能理解：

- 这个产品判断的是家庭财务状态和未来目标是否可支撑。
- 用户需要输入资产、负债、收入、支出和目标。
- 输入这些内容之后，会得到一份可继续追问的 AI 分析。

### 当前问题

- 首页价值表达还不够直给，用户可能不知道“能躺了吗”具体在判断什么。
- 输入更像字段表单，而不是自然问题流。
- 用户不知道每一项输入会影响后续哪类判断。
- 输入完成前，价值反馈偏弱。

### 改造方向

1. 第一屏说明产品定位
   - 用一句话说明：这是一个家庭财务与人生目标压力测试工具。
   - 避免像记账软件、理财销售页或泛泛的 AI 聊天入口。

2. 输入流程按问题组织
   - 你是为自己规划，还是和伴侣/夫妻一起规划？
   - 你有哪些家庭责任？
   - 你主要生活在哪类城市？
   - 你现在有多少灵活资产和固定资产？
   - 你现在还有多少贷款或负债？
   - 每年收入、支出大概是多少，预计还会工作多少年？
   - 未来有哪些重要目标？

3. 每一步给出价值提示
   - 资产：用于判断安全垫和资源总量。
   - 可动用资产：用于判断短中期抗风险能力。
   - 负债：用于判断财务脆弱度。
   - 收支：用于判断未来目标能否持续推进。
   - 目标：用于判断“能不能躺”到底是在覆盖什么。

4. 保留低门槛输入
   - 允许估算。
   - 允许跳过非核心项。
   - 默认值必须谨慎，不能让用户误以为系统已经知道真实情况。

### 第一版交付

- 首页文案重写。
- 输入流程重新分组。
- 每个输入组增加一句价值提示。
- 生成一份结构化家庭财务画像，作为 AI 分析输入。

## 分支二：AI 分析输出

### 目标

让最终结果页主要呈现大模型输出，而不是把固定规则包装成 AI。

固定规则仍然可以存在，但它的角色变为：

- 计算基础指标。
- 组织结构化上下文。
- 给大模型提供判断材料。

用户看到的核心分析，应由大模型基于用户数据生成。

### 当前问题

- 输出内容容易千篇一律。
- AI 的参与感不强，更像固定模板。
- 用户输入变化后，结果差异不够明显。
- 结果页同时存在规则结论和 AI 表达，体验上可能重复。

### 改造方向

1. 固定规则退到幕后
   - 前端仍计算资产、负债、结余、覆盖率、目标缺口等指标。
   - 这些指标不再全部直接展示，而是作为 Prompt 上下文。

2. 大模型生成主要分析
   - 当前结论。
   - 最大风险。
   - 为什么会这样。
   - 接下来三步。
   - 可以继续追问的问题。

3. Prompt 强化差异化
   - 要求模型引用用户的具体数字和目标。
   - 要求模型区分资产多但不可动用、收入高但支出高、目标过重、负债过高等不同情况。
   - 禁止只输出泛泛建议。

4. 输出结构可控
   - 优先让云函数要求模型返回结构化 JSON。
   - 前端按固定区块展示 JSON 内容。
   - 如果 JSON 解析失败，再降级展示 Markdown 文本。

### 第一版交付

- 设计 AI 分析 Prompt。
- 云函数支持 `analysis` 类型请求。
- 结果页改为展示 AI 生成内容。
- 保留“继续问 AI”作为追问入口。

## 两条分支的交接点

两条线之间需要一个稳定的数据结构：家庭财务画像。

第一版 MVP 画像控制在 2 分钟内可完成，用户侧只问 7 组问题，系统侧整理成结构化数据后传给大模型。

```json
{
  "profile": {
    "planningScope": "single",
    "familyResponsibilities": ["children", "education", "elderCare"],
    "cityTier": "newFirstTier"
  },
  "assets": {
    "liquidAssets": 300000,
    "fixedAssets": 1800000,
    "totalAssets": 2100000
  },
  "liabilities": {
    "totalLiabilities": 900000
  },
  "cashflow": {
    "annualIncome": 450000,
    "annualExpense": 280000,
    "expectedWorkYears": 15
  },
  "goals": [
    {
      "type": "education",
      "name": "孩子教育",
      "amount": 800000,
      "targetYear": 12,
      "priority": "need"
    },
    {
      "type": "semiRetirement",
      "name": "降低工作强度",
      "amount": 2500000,
      "targetYear": 15,
      "priority": "want"
    }
  ],
  "computed": {
    "netWorth": 1200000,
    "annualSurplus": 170000,
    "liquidAssetRatio": 0.14,
    "debtRatio": 0.43,
    "annualSurplusRate": 0.38
  }
}
```

分支一负责把用户输入整理成这份画像。

分支二负责让 AI 基于这份画像生成判断。

### MVP 画像字段约定

用户侧问题：

1. 你是为自己规划，还是和伴侣/夫妻一起规划？
2. 你有哪些家庭责任？可多选：有孩子、需要考虑教育、需要赡养老人、其他。
3. 你主要生活在哪类城市？一线城市、新一线/强二线、普通二线、三四线及以下。
4. 你现在大概有多少灵活资产和固定资产？
5. 你现在大概还有多少贷款或负债？
6. 你家一年大概收入和支出是多少？预计还会工作多少年？
7. 你未来有哪些重要目标？可多选：买房/换房、孩子教育、养老储备、提前退休/降低工作强度、旅行/生活方式、其他。每个目标补充金额、几年后发生、必须要或想要。

字段价值：

- `planningScope` 判断分析口径是个人还是家庭。
- `familyResponsibilities` 判断教育、养老、家庭责任压力。
- `cityTier` 辅助判断生活成本和目标压力。
- `liquidAssets` 判断短中期抗风险能力。
- `fixedAssets` 判断账面资产与可动用资产是否分化。
- `totalLiabilities` 判断负债压力。
- `annualIncome` 和 `annualExpense` 计算年度结余。
- `expectedWorkYears` 判断未来现金流还能持续多久。
- `goals` 让“能不能躺”的判断有明确目标，而不是泛泛看财务健康。

## 推荐执行顺序

1. 先定义家庭财务画像字段。
2. 重写首页和输入流程，让用户能自然产生这份画像。
3. 设计 AI 分析 Prompt。
4. 改造结果页，只展示 AI 生成的核心判断。
5. 用 3-5 个典型家庭样本测试输出差异。

## 测试样本

后续至少准备这些样本：

- 资产高但大多是房产，现金少。
- 收入高但支出也高，结余低。
- 资产一般但负债低，目标温和。
- 有大额教育、养老、买房目标叠加。
- 没有明确目标，只想知道现在是否安全。

这些样本用来验证：

- 用户输入是否顺畅。
- AI 是否能说出不同判断。
- 输出是否具体、有帮助、不过度吓人。
- 用户是否愿意继续追问。

## 当前结论

接下来不要继续堆更多页面。先把“输入有意义”和“AI 输出有洞察”做扎实。

这两件事做好之后，Demo 才会从“能跑”变成“有产品价值”。
